# Publio AI 编辑工作台设计稿

## 背景

当前项目已经具备基础能力：

- Markdown 编辑与发布
- AI 行业 RSS 聚合
- 一键分发到多个平台

但产品体验存在两个核心问题：

1. `AI 行业资讯抓取` 无法稳定发现真正值得写的热点，输出更像 RSS 列表，而不是内容编辑候选池。
2. `UI 与产品结构` 缺乏统一的工作台感，首页、AI 新闻页、编辑器预览区呈现出割裂的视觉和交互体验。

本次设计目标不是小修小补，而是重定义该产品的主价值：

> Publio 不只是“抓新闻 + 自动写稿”的工具，而是一个围绕 AI 热点选题、研究底稿和多平台发布的编辑工作台。

## 目标

- 把 AI 新闻模块从“新闻列表”升级为“候选题编辑台”
- 把成文逻辑从“自动生成文章”升级为“生成研究底稿”
- 让用户先完成选题判断，再决定是否进入写作
- 统一首页、AI 新闻页、编辑器与发布流程的产品语言
- 提升整体视觉专业度与产品一致性

## 非目标

- 第一阶段不做复杂的用户偏好学习系统
- 第一阶段不引入真正的 AI 大模型摘要编排链路
- 第一阶段不做完整账号体系或团队协作系统
- 第一阶段不追求“自动生成可直接发布的高质量长文”

## 用户目标

目标用户是以中文内容平台为主要阵地的 AI 内容创作者或编辑。其核心需求不是获取最全资讯，而是更快回答以下问题：

- 今天最值得写的 AI 话题是什么
- 哪些话题虽然不是刚发生，但仍然有发酵空间
- 这件事为什么重要
- 它影响了谁
- 如果要写，应该从什么角度切入

该用户希望系统给出一组高质量候选，由自己最终拍板，而不是完全依赖系统自动决定最终选题。

## 核心设计原则

### 1. 编辑优先，而非资讯堆叠

系统输出应该服务于编辑判断，而不是服务于“展示抓到了多少新闻”。

### 2. 候选优先，而非自动成稿

首页级目标不是自动写出文章，而是提供足够有价值的候选题与研究底稿，帮助用户快速进入创作。

### 3. 结构化判断优先，而非模板化摘要

每条热点的价值在于“为什么重要、影响了谁、能怎么写”，而不是生成一些通用段落。

### 4. 产品统一优先，而非页面各自成立

所有页面都应体现“内容编辑工作台”的一致结构、层级、视觉和状态反馈。

## 一、资讯发现与候选池机制

### 当前问题

当前 `ai-news` 逻辑主要依赖多条 Google News 搜索 RSS：

- 发现范围受限
- 结果容易偏成“搜索命中新闻”
- 同一事件的多篇报道容易挤占候选位
- 排序维度过于依赖关键词和发布时间
- 对“中文内容传播里是否值得写”缺乏判断能力

因此最终输出更接近“AI 新闻流”，而不是“编辑候选池”。

### 新的数据流

新的新闻发现链路定义为：

`多源抓取 -> 清洗标准化 -> 聚类去重 -> 多维评分 -> 候选池分层 -> 研究底稿生成`

### 1. 多源抓取

第一阶段抓取源至少拆成三类：

- `权威科技媒体源`
  作用：保证事实密度和报道质量
- `官方发布源`
  作用：保证事件源头可信，可追溯到公司、产品或项目公告
- `中文传播敏感源`
  作用：辅助判断一个话题是否具备中文内容场中的传播潜力

第一版可以仍以 RSS 或公开可抓取 feed 为主，但架构上应把“源类型”建模出来，而不是把所有来源都视为平级新闻项。

### 2. 清洗标准化

抓取结果进入统一标准结构，至少包含：

- 标题
- 原始链接
- 来源名称
- 来源类型
- 发布时间
- 摘要
- 是否为官方源
- 抓取时间

这一步的目标是为后续聚类和评分提供稳定输入，而不是在此阶段生成文案。

### 3. 聚类去重

系统最终不再以“新闻条目”为核心展示单位，而以“话题簇”为核心单位。

同一事件被多个来源报道时，需要尽量归并为一个候选题，候选题内部保留：

- 主标题
- 关联报道数量
- 关键来源列表
- 最早发布时间
- 最新发布时间

聚类的判断依据应包括：

- 标题相似度
- 公司 / 产品 / 事件关键词重合度
- 发布时间窗口接近度
- 链接域名与来源类型辅助判断

### 4. 多维评分

每个候选题至少拆成 4 个分数：

- `新鲜度`
  衡量是否足够新，适合作为“今天能发”的话题
- `行业影响`
  衡量该事件影响面，是否改变产品、模型、算力、商业化、政策或竞争格局
- `传播势能`
  衡量该事件是否具备中文内容传播中的切题性、标题感和讨论潜力
- `可信度`
  衡量来源质量，是否有官方依据或高质量媒体交叉确认

系统最终排序不再只输出一个总分，而应保留各维度结果，以支持 UI 解释“为什么它排在这里”。

### 5. 候选池分层

候选池默认拆成两层：

#### `今天能发`

定位：过去 24 小时内适合直接切入的热点。

入池偏好：

- 新鲜度高
- 行业影响明确
- 传播势能高
- 信息可信

#### `还能追`

定位：过去 3 天内仍在发酵、适合做延展解读的主题。

入池偏好：

- 虽非最新，但后续影响仍在展开
- 具备背景延展空间
- 适合做趋势、竞争、影响分析

### 6. 候选卡片输出结构

每个候选题在列表上不直接展示长摘要，而展示编辑判断摘要。建议字段为：

- 候选标题
- 话题标签
- 发布时间或发酵窗口
- 关键来源数 / 关联报道数
- `为什么值得写`
- `影响了谁`
- 推荐切入角度

其核心是帮助用户快速判断“要不要点开看底稿”，而不是先看一大段正文。

## 二、研究底稿生成机制

### 当前问题

当前逻辑倾向于输出接近文章格式的长文块：

- 信息结构不稳定
- 容易产生空泛套话
- 难以支持编辑快速判断
- 直接成稿能力又不足，导致既不像工作底稿，也不像成熟文章

### 新的输出目标

点击候选题后，系统默认进入 `研究底稿`，而不是直接进入“自动生成长文”。

研究底稿的目标是：

- 帮助用户快速判断是否值得写
- 帮助用户理解为什么重要
- 帮助用户识别影响对象
- 帮助用户决定可用写法
- 为后续进入写作台提供结构化素材

### 研究底稿固定结构

每条底稿建议固定为 6 个模块：

#### 1. 事件是什么

用一段极短的事实描述回答：

- 谁做了什么
- 在什么时间点发生
- 最值得注意的动作是什么

#### 2. 为什么重要

这是底稿中的最高优先级模块，应给出明确编辑判断，而不是复述摘要。要尽量回答：

- 它为何不是普通动态
- 它改变了哪类竞争关系、预期或行业节奏
- 它为什么值得写成内容

#### 3. 影响了谁

按对象拆分，而不是笼统说“对行业有影响”。候选对象包括：

- 模型厂商
- AI 应用公司
- 开发者
- 企业客户
- 投资市场
- 内容创作者

#### 4. 推荐写法

给用户几个可用的中文内容切口，而不是直接代写文章。每个切口包含：

- 切口名称
- 一句说明其为什么适合写

可选切口包括：

- 新闻解读型
- 趋势判断型
- 公司竞争型
- 机会 / 风险型

#### 5. 延展背景

给出继续写作所需的上下文，比如：

- 与最近哪些连续动作有关
- 与哪条主线趋势相关
- 可对比的公司、模型、政策或竞品

#### 6. 原始依据

保留来源列表，并尽量区分：

- 官方发布
- 主流媒体报道
- 其他转载或聚合引用

### 产品行为变化

研究底稿生成后，页面默认提供 3 个动作：

- `查看研究底稿`
- `加入写作台`
- `按这个角度生成初稿`

其中：

- `查看研究底稿` 是首要动作
- `加入写作台` 是次要动作
- `按这个角度生成初稿` 是更深一层动作，不应成为首页默认主按钮

### 设计结论

产品应从“自动写稿工具”转向“编辑辅助工具”，先服务判断，再服务写作。

## 三、产品结构重组

### 当前问题

当前项目的信息架构偏“页面并列”：

- 首页负责写作
- AI 新闻页负责资讯
- 设置页负责配置

它们各自可用，但缺乏统一工作流，因此用户在产品中的角色切换不够清晰。

### 新的一级结构

我建议把产品重组为三个一级工作区：

#### 1. 选题台

定位：从热点中筛选出值得写的候选题。

核心内容：

- 今天能发
- 还能追
- 候选卡片列表
- 候选详情抽屉 / 侧栏
- 研究底稿面板

#### 2. 写作台

定位：将候选底稿转化为可编辑内容，并完成多平台分发。

核心内容：

- 标题区
- Markdown 编辑区
- 素材引用区
- 发布态预览区
- 平台分发区
- 发布结果反馈区

#### 3. 设置与平台

定位：账号配置、平台授权和发布参数设置，不干扰主流程。

### 关键工作流

产品的理想主流程定义为：

`进入选题台 -> 浏览候选池 -> 打开研究底稿 -> 判断值得写 -> 加入写作台 -> 编辑 -> 预览 -> 发布`

这条主流程必须在信息架构、按钮文案和页面切换上都足够清晰。

## 四、UI 统一设计方向

### 当前问题

当前界面存在以下割裂：

- 首页是暖色编辑台风格
- 预览区切换成深色内容容器风格
- AI 新闻页是另一套资讯卡片风格
- 大量组件在层级、阴影、边框、色彩上没有统一系统

这会让产品显得像多个页面拼接在一起，而不是一个成熟系统。

### 推荐视觉方向

建议整体走 `专业 SaaS + 编辑部气质` 的混合方向。

这意味着：

- 保留内容工具的专业感和信息密度
- 减少过强的装饰混搭
- 通过排版、留白、层级和状态反馈体现高级感
- 强化“编辑工作台”而不是“营销落地页”气质

### 视觉系统原则

#### 1. 统一色彩系统

- 主体基调使用克制的中性色
- 保留一个稳定的品牌强调色
- 强调色用于主操作、标签、高亮状态
- 避免首页暖色、预览区深色、资讯页浅色之间的割裂切换

#### 2. 统一空间与卡片系统

- 统一圆角尺度
- 统一边框颜色与阴影级别
- 统一卡片内边距和模块间距
- 统一页面宽度、栏位和留白体系

#### 3. 统一排版系统

- 为页面标题、副标题、标签、正文、辅助文案建立稳定层级
- 英文小标题与中文正文需统一视觉节奏
- 减少视觉装饰型 uppercase 的滥用，保留必要的品牌识别即可

#### 4. 统一交互反馈

- 统一按钮优先级
- 统一加载、错误、成功状态表达
- 统一“列表 -> 详情 -> 动作”的交互路径

### 页面级改造建议

#### 选题台

- 列表以候选池为核心，而不是新闻流
- 使用右侧详情抽屉或固定详情面板承载底稿
- 卡片突出编辑判断，而不是长摘要

#### 写作台

- 减弱“庞大编辑器直接压上来”的压迫感
- 明确标题区、正文区、素材区、平台区四层关系
- 预览区风格与整体系统统一，不再像另一套产品

#### 发布区

- 更像工作流状态面板
- 强化平台状态、失败原因和下一步动作
- 避免一次性结果感

## 五、数据与模块边界建议

为了支持上述产品变化，`ai-news` 模块需要从“单条新闻列表”转向“候选题数据模型”。

建议数据结构分成三层：

### 1. 原始抓取项

对应每个来源抓到的新闻条目。

### 2. 候选题簇

对应聚类后的可编辑候选题，是选题台中的主对象。

### 3. 研究底稿

对应围绕某个候选题生成的结构化编辑底稿。

这样做的好处是：

- 抓取逻辑、聚类逻辑、展示逻辑彼此解耦
- 候选题可以承接后续用户反馈与偏好学习
- 写作台可以明确引用“某个候选题”而不是一堆散乱新闻项

## 六、错误处理与降级策略

候选池能力不能依赖“所有源都成功抓取”。建议设计以下降级方式：

- 单个源抓取失败时，不阻断整体候选池生成
- 聚类失败时，至少回退到基础列表视图
- 某候选缺乏足够来源时，降低可信度评分而不是直接删除
- 研究底稿生成信息不足时，明确提示“需要补充原文确认”，而不是生成装作完整的判断

## 七、测试与验证重点

后续实现阶段需要重点验证以下行为：

- 同一事件是否能被合理聚类
- 排序结果是否更接近“值得写”的候选，而不是简单最新资讯
- “今天能发 / 还能追”的分层是否符合预期
- 研究底稿是否能稳定回答“为什么重要、影响了谁”
- 用户能否自然完成“从候选题进入写作台”的主流程
- 新 UI 是否在首页、AI 新闻页、编辑器和发布区之间保持统一

## 八、分阶段实施建议

建议实现顺序如下：

### 阶段 1：重构 AI 新闻数据模型

- 引入抓取项、候选题簇、底稿三层结构
- 重写聚类、排序与候选池生成逻辑

### 阶段 2：重构 AI 新闻页为选题台

- 上线“今天能发 / 还能追”结构
- 增加候选详情与研究底稿展示

### 阶段 3：重构写作台衔接

- 支持从候选题进入编辑器
- 把研究底稿中的结构化信息带入写作台

### 阶段 4：统一 UI 系统

- 收敛色彩、卡片、间距、排版、状态与交互体系
- 统一首页、选题台、写作台、发布区

### 阶段 5：为偏好学习预留接口

- 为未来“值得写 / 不值得写”的用户反馈闭环预留数据入口
- 但第一阶段不做完整偏好学习逻辑

## 设计结论

本次设计的核心不是“把资讯页做得更好看”，而是让 Publio 的核心价值从“多平台发布工具”升级为“AI 热点选题与创作工作台”。

它的主张应变为：

- 不只是帮用户写
- 更先帮用户找出真正值得写的题
- 再用研究底稿帮助用户快速进入创作

这也是后续 UI 和功能统一的基础。
